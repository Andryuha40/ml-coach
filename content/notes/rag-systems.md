# RAG (Retrieval-Augmented Generation)

Конспект собран целиком из одного источника — записи лекции «13. Запись
лекции, 25 апреля 2026». Это часть уникального для курса блока про LLM/
агентов/RAG, которого нет в репозитории Евгения Паточенко — отдельного
`repo_lesson` для этой темы не существует, поэтому лекция — единственный
источник знаний.

## Повторение: этапы обучения больших языковых моделей

1. **Претренинг (pretraining)** — модель обучается на огромном объёме
   данных (условно — весь доступный текстовый интернет). На этом этапе
   модель учится «целевому распределению языка», то есть учится говорить:
   правдоподобно продолжать текст. Она не решает конкретные задачи, а
   лишь предсказывает наиболее вероятное продолжение — за счёт огромного
   количества вычислений и данных в её весах оседают разнообразные знания
   о мире. Если такой «сырой» модели задать конкретную задачу (например,
   арифметическую), она не решит её, а просто продолжит текст в
   правдоподобном, но бессмысленном для решения ключе.
2. **SFT (Supervised Fine-Tuning)** — модель дообучают генерировать текст
   в соответствии с конкретными задачами: вести диалог, суммаризировать
   текст, писать код и т. д. Поскольку текстом можно описать практически
   любую задачу (в том числе задачи классического ML), после SFT модель
   начинает отвечать на прикладные вопросы уже осмысленно.
3. **Alignment (RLHF)** — ещё более тонкая настройка модели под
   предпочтения пользователей, основанная на модели вознаграждения
   (reward model), которая награждает или наказывает модель за то или
   иное поведение. Построена на подкреплённом обучении (reinforcement
   learning).

## Зачем нужен RAG

RAG (Retrieval-Augmented Generation) — способ отвечать на запрос
пользователя, используя дополнительный внешний источник данных, а не
полагаясь исключительно на внутренние знания модели. Технически это
генеративный ответ по внешним источникам: хороший поиск (классический
или векторный, по векторной базе данных) + хороший генератор.

Почему бы не поместить все знания о мире прямо в модель на этапе
претренинга или SFT? Есть большой класс запросов, на которые модель,
обученная и зафиксированная один раз, ответить не может:

- **Персональные, специфичные для пользователя данные.** Например,
  «сколько дней отпуска у меня осталось в этом году» — модель ничего не
  знает ни о конкретном пользователе, ни об актуальном балансе его
  отпускных дней, потому что эта информация постоянно меняется и не
  может быть «зашита» в модель на этапе обучения.
- **Актуальные, часто обновляющиеся данные.** Погода, результаты
  футбольных матчей, текущая цена барреля нефти и т. п.
- **Специфичные, критичные с точки зрения точности запросы**, где
  неверный ответ (галлюцинация) может иметь серьёзные последствия —
  например, «какой протокол лечения грибковой пневмонии». Здесь нужен
  надёжный ответ, строго опирающийся на источники — желательно со
  ссылками на них и возможностью верификации.

**Проблема галлюцинаций.** Галлюцинации у языковых моделей — системная
особенность: их природа стохастическая, и абсолютно исключить
галлюцинации нельзя. RAG сам по себе не устраняет вероятность
галлюцинации полностью, он лишь снижает её по сравнению с ответами модели
«из памяти». Дополнительно на снижение галлюцинаций сильно влияет этап
Alignment/RLHF.

## Как быстро собрать базовую версию RAG

RAG состоит из двух базовых компонент: **поиска** и **генератора**. Схема
работы:

1. Есть запрос пользователя и внешняя база знаний (внешняя — по отношению
   к знаниям самой LLM; для конкретной компании эта база может быть
   внутренней, но «внешней» с точки зрения модели).
2. Собирается база документов, документы нарезаются на кусочки (чанки) —
   по количеству символов, токенов, абзацам, предложениям или
   параграфам. Нарезка делается с перекрытием (оверлапом) между соседними
   чанками, чтобы не терять информацию на границах.
3. Чанки складываются в векторную базу данных. Важно сразу продумать
   регулярное обновление этой базы (например, через периодический
   парсинг источника данных или API), потому что база знаний со временем
   меняется.
4. По запросу пользователя поисковая компонента возвращает top-k наиболее
   релевантных (семантически близких) документов/чанков.
5. Генератор (LLM, например, через API одной из open-source моделей)
   получает промт, в который подставляются запрос и найденные документы,
   и генерирует ответ.

Такое базовое решение можно собрать очень быстро — «за чашку кофе», в
течение одного вечера, при некоторой сноровке — используя готовые
фреймворки: LangChain — для работы с текстами, sentence-transformers —
для готовых предобученных эмбеддинговых моделей, любая доступная LLM
(открытая или через API) — как генератор.

## Методология итеративного улучшения RAG

Базовое решение (RAG-baseline), собранное «за вечер», обычно сразу
порождает целый ворох проблем: галлюцинации, отсутствие ответа там, где
он должен быть, неверные или неполные ответы и т. д. Просто дёргать
параметры фреймворка наугад — плохая стратегия: одни проблемы уходят,
вылезают другие. Нужен системный подход, аналогичный классическому ML —
сначала строится базовая версия, а затем итеративно, шаг за шагом,
происходит диагностика и исправление проблем:

1. **Честная разметка существующих ответов.** Берётся честный срез
   реального потока запросов (или того, что предполагается получать в
   проде). Срез размечается — желательно силами нескольких участников
   (разработчики, QA, продакт-менеджеры, маркетологи, аналитики,
   руководители — подходят все, кто понимает продукт), чтобы получить
   объективную картину.
2. **Локализация проблем и подсчёт процента «хороших» ответов.** По
   итогам разметки получается процент ответов, которые устраивают, и
   процент, требующий доработки.
3. **Категоризация и приоритизация проблем.** Оставшиеся «плохие» ответы
   разбиваются по типам проблем в некотором процентном соотношении,
   после чего проблемы приоритизируются — по степени влияния на качество.
   Начинают решать с самых массовых проблем, к редким возвращаются
   позже.
4. **Итеративное исправление.** Проблема решается — качество замеряется
   заново — переходим к следующей проблеме. Так по кругу.

Почти все проблемы, за редким исключением, не связаны друг с другом:
решение одной практически не влияет на другие части системы, поэтому их
можно чинить по отдельности. Как правило, при анализе выясняется, что
часть проблем приходится на компонент поиска, часть — на компонент
генератора (например, условно 60% на поиск, 40% на генератор).

**Типовые причины проблем:**
- **Достоверность** — модель пишет то, чего нет в источниках
  (галлюцинирует).
- **Полнота** — ответ есть, но он неполный, не отвечает на вопрос
  целиком.
- **Нет ответа, хотя знание в базе есть** — модель не смогла извлечь
  нужную информацию из документов и сгенерировать корректный ответ.
- **«Отмороженные» ответы** — модель вместо ответа переспрашивает
  пользователя или не отвечает по существу, хотя вся необходимая
  информация в контексте есть.

## Проблема: нет релевантных документов в выдаче (проблема поиска)

Известно, что нужный документ есть в базе, но при поиске по запросу
пользователя он не попадает даже в top-30/40/50 выдачи:

- Если используется классический (текстовый) поиск — проблема, скорее
  всего, в самом поисковом механизме.
- Если используется векторный поиск — проблема, скорее всего, в
  эмбеддере. Нужно проверить, обучен ли используемый эмбеддер именно на
  задачу поиска документов по запросу (retrieval), а не просто «хороших
  эмбеддингов вообще» — модель, не заточенная под конкретную задачу,
  часто плохо работает именно на retrieval.
- Решение — подобрать другую, более подходящую эмбеддинговую модель, либо
  дообучить имеющуюся под свою базу знаний / поисковую выдачу.
- Частая и малопонятная по причинам проблема: локально (на небольшом
  наборе) поиск работает хорошо, а на всей базе — плохо. Здесь может
  помочь **гибридный поиск** — сочетание текстового поиска (для
  кандидатов) и векторного поиска.

## Проблема: релевантные документы есть, но не в топе выдачи (реранкинг)

Документы, содержащие нужную информацию, попадают в выдачу, но не в топ,
а «размазаны» где-то по списку. Поскольку контекстное окно модели
ограничено (и это ограничение реально болезненно даже у крупных компаний
с большими бюджетами), нельзя просто засунуть всю выдачу в контекст —
нужно, чтобы релевантные документы оказались наверху, а нерелевантные —
внизу.

Для этого используется техника **реранкинга** (re-ranking) с помощью
дополнительной модели — **реранкера**. Реранкер принимает на вход пару
(запрос, документ) и предсказывает скор их соответствия. Взяв, например,
top-10 документов из первичной выдачи, можно посчитать скор для каждой
пары запрос-документ и пересортировать документы по убыванию скора. В
качестве реранкера можно использовать готовые модели (например, из
семейства BERT/кросс-энкодеров) — их довольно много, при необходимости их
также можно дообучать под свою задачу.

Дополнительная техника: если после реранкинга у части чанков скор
оказывается ниже некоторого порога (трешхолда), такие чанки можно просто
отбрасывать — они слаборелевантны и только зря занимают место в контексте.
Это, как и многие подобные приёмы, «может сработать, а может не
сработать» — нужно проверять на своих данных.

**BERT vs GPT (справочно).** BERT — не одна модель, а целое семейство
(RoBERTa, ALBERT и другие, порядка полусотни разновидностей), которое
было очень популярно в 2018–2019 годах. Сейчас семейство BERT
используется гораздо более точечно — для узких задач (классификация,
детекция), потому что большие языковые модели решают почти все подобные
задачи лучше, просто с гораздо большими вычислительными затратами.
Поэтому там, где нужно закрыть небольшую, узкую задачу, «стрелять» по ней
большой LLM невыгодно — здесь как раз может пригодиться небольшая
специализированная модель вроде BERT-классификатора, которую к тому же
можно развернуть локально.

## Проблемы качества документов в базе знаний

- **Низкое качество отдельных документов** — например, из-за ошибок
  распознавания (OCR картинок) или неудачного парсинга страниц, либо
  изначально «мусорных» документов. Решение — обучать или использовать
  готовые модели-классификаторы «мусорности» (FastText, BERT-
  классификаторы), либо эвристики: доля неалфавитных/спецсимволов, доля
  служебных слов без смысловой нагрузки, повторяющиеся символы и т. п.
- **Дубли и почти-дубли документов.** Если в релевантную выдачу попадает
  несколько почти одинаковых документов, они занимают весь топ и обедняют
  ответ. Бороться можно, заводя качественную эмбеддинговую модель,
  умеющую вычленять семантику и искать семантические дубли, и держать в
  базе только уникальные документы.
- **Тематически бесполезные документы** — например, полупустые
  страницы-заготовки. Детектируются эвристиками (структурные шаблоны),
  моделями-классификаторами, либо прогоном через LLM с промтом на
  верификацию (дороже по вычислениям).
- **Сгенерированные («синтетические») источники.** При классическом
  сборе данных из интернета в базу может попасть много LLM-
  сгенерированного контента — от такого стоит избавляться: информация в
  нём может быть недостоверной, плюс это несёт репутационные риски.
- **Кастомные форматы документов** (PDF, презентации, DOCX и т. д.) — для
  качественного извлечения данных лучше использовать под каждый формат
  отдельный, специально настроенный парсер.
- **Устаревшие документы.** Базы знаний со временем устаревают, особенно
  критично для доменов вроде законодательства. Способы борьбы: правило
  автоматического удаления устаревших документов (по TTL или по факту
  появления новой версии источника); использование даты создания/
  добавления документа как части контекста, передаваемого генератору (чтобы
  модель сама учитывала давность документа). В отдельных случаях (например,
  законодательство, где важно различать нормативную базу разных периодов)
  может понадобиться хранить сразу несколько версий одного и того же
  документа для разных временных промежутков.

## Проблемы с разбиением на чанки

Проблемы с чанками обычно проявляются как проблемы с поиском: в контекст
попадает кусочек документа, релевантный по теме запроса, но обрывающийся
на важной информации — модель теряет часть данных, необходимых для
полного ответа.

Основные стратегии формирования чанков:

1. **Fixed-size chunking** — нарезка по фиксированному размеру (например,
   по количеству токенов) с обязательным перекрытием (overlap) между
   соседними чанками (классическое соотношение — например, размер чанка
   1000 токенов, перекрытие 100). Оптимальный размер подбирается
   эмпирически: зависит от скорости, размера контекста модели, числа
   документов и итогового качества ответа. Компромисс: чем крупнее чанк,
   тем больше контекста модели он занимает и тем медленнее всё работает,
   но и тем больше полезной информации в него помещается.
2. **Обогащение чанков метаинформацией.** В чанк можно добавлять
   заголовок, автора документа, теги и т. д. (например, для медицинского
   ассистента — данные об авторе статьи как индикатор достоверности).
3. **Разбиение по структуре документа.** Если в домене есть чёткая
   структура (законодательные акты, разбитые на статьи и пункты; внутренние
   вики-страницы с разделами), можно и нужно резать по этой структуре.
   Минусы: не всегда структура отражает смысл; встречаются слишком крупные
   разделы, которые всё равно приходится дробить дополнительно, стараясь
   сохранить общую информацию (например, название подраздела) в каждом
   получившемся под-чанке.
4. **Кластеризация на основе эмбеддингов.** Документ разбивается на
   предложения, каждое предложение векторизуется лёгкой эмбеддинговой
   моделью. Предложения последовательно добавляются в кластеры: для
   каждого нового предложения считается косинусное расстояние до
   усреднённого эмбеддинга существующего кластера — если близко,
   предложение добавляется в кластер, если нет — создаётся новый.
5. **Формирование чанков в рантайме («на лету»).** Вместо того чтобы
   заранее (в офлайне) резать все документы на чанки, релевантный документ
   находится целиком через поиск, а уже после этого из него в рантайме
   нарезаются чанки и скорятся относительно конкретного запроса
   пользователя.
6. **Чанкинг с помощью LLM.** Документ прогоняется через LLM с просьбой
   суммаризировать содержимое (не меняя сути), а получившаяся
   суммаризация разбивается на несколько частей — эти части складываются
   в базу как чанки.

## Проблемы с непонятными запросами пользователей

Запросы пользователей часто сформулированы небрежно, с опечатками,
обрывочно. Способы борьбы:

- **Исправление опечаток** — давно отработанные модели (аналог
  автокоррекции на смартфонах).
- **Перефразирование запроса.** Плохо составленный запрос
  переформулируется в «хороший» с помощью модели, вместо того чтобы
  переспрашивать пользователя. Действовать нужно деликатно — неудачное
  перефразирование может исказить исходный смысл запроса.
- **Генерация дополнительных запросов (multi-query).** К исходному
  запросу генерируется несколько похожих вариаций, для каждой делается
  отдельная выборка релевантных чанков, результаты объединяются.
- **Извлечение ключевых слов** — дополнительно расширяет поисковый
  запрос.

## Итог по методологии улучшения RAG

После создания базовой версии (RAG-baseline): анализируем проблему → если
качество плохое (а поначалу оно почти всегда где-то плохое) → сводим
анализ к крупным кластерам проблем, которые важно чинить в первую очередь.
Если нет релевантных документов в выдаче — работаем с эмбеддером/поиском.
Если документы не попадают в топ — добавляем реранкер. Если в базе мусор
— фильтруем эвристиками и/или дополнительными моделями, донастраиваем
чанкинг. Если приходят непонятные запросы — занимаемся рефрейзингом либо
генерацией дополнительных запросов.

Большинство перечисленных техник не требуют сверхсложных инженерных
решений — это относительно небольшие, доступные приёмы, которые вполне
можно реализовать в одиночку.

## Практика: построение RAG-системы на отзывах с КиноПоиска

Цель — построить простого ассистента, отвечающего на произвольные
вопросы о фильмах на русском языке, используя в качестве базы знаний
отзывы с КиноПоиска.

**Данные.** Датасет с отзывами загружается с Hugging Face Hub (аналог
GitHub для моделей и датасетов в Deep Learning) через библиотеку
`datasets` и функцию `load_dataset`. После предобработки (замена мусорных
символов, лишних пробелов через `map`) — 36 591 запись. Ключевые поля:
название фильма и содержимое отзыва (текст + оценка пользователя).

**Базовая (без RAG) LLM.** Генератор — небольшая открытая модель Qwen на
1,5 млрд параметров, инструктивная версия из GPT-подобных архитектур,
мультиязычная. Загружается через `pipeline` из библиотеки `transformers`
(обвязка над PyTorch), с указанием типа тензора (float16/float32) и
устройства.

Тестовый запрос без RAG: «В каких 5 фильмах играл Роберт Де Ниро?».
Параметры генерации: 256 токенов, температура и top_p (параметры
сэмплирования, отвечающие за «творчество»/разнообразие модели — сужают
или расширяют распределение возможных ответов). Результат — модель выдала
список, часть фильмов оказались полностью выдуманными. Причина —
недостаток данных на русском языке про кино в обучающей выборке модели:
модель всё равно правдоподобно генерирует ответ, даже не обладая нужными
знаниями, то есть галлюцинирует.

**Построение векторной базы знаний:**
1. Выбирается эмбеддинговая модель — большая мультиязычная модель на
   основе трансформера, загружается через `sentence-transformers`.
2. Выбирается векторная база данных — Qdrant. Через клиент Qdrant
   создаётся коллекция (`create_collection`) с указанием размера вектора
   (должен совпадать с размерностью эмбеддинговой модели), метрики
   близости (косинусное расстояние) и флага сохранения на диск.
3. Документы (отзывы) режутся на чанки, поскольку они слишком длинные,
   чтобы целиком поместиться в контекст модели. Используется рекурсивный
   сплиттер из LangChain (`RecursiveCharacterTextSplitter`) — один из
   самых «безобидных» методов, чаще работает, чем нет.
4. Чанки кодируются эмбеддинговой моделью (`encode`), с указанием
   нормализации векторов и устройства (GPU).
5. Полученные векторы вместе с метаданными (текст отзыва, год выхода
   фильма и т. д.) загружаются в коллекцию Qdrant как «точки» (points),
   каждая содержит id, вектор и полезную нагрузку (payload).

**Проверка семантического поиска (без генератора).** Запрос «Роберт Де
Ниро» кодируется той же эмбеддинговой моделью, находятся 5 ближайших
документов — результат оказался достаточно точным. В узких доменах вроде
«отзывы о кино» метрики поиска обычно получаются заметно выше, чем в
реальных прикладных задачах.

**Построение генератора RAG.** Промт: «Ты — русскоязычный эксперт в
области кинематографа. У тебя есть доступ к набору отзывов о фильмах,
используй их, чтобы полно и точно ответить на следующий вопрос...
Убедись, что ответ подробный, конкретный и непосредственно касается
вопроса, не добавляя информацию, которая не подтверждается предоставленными
отзывами». Ключевой элемент промта — явное ограничение отвечать только на
основе предоставленного контекста, без привлечения других источников:
именно это ограничивает галлюцинации модели.

Присвоение модели роли (в промте — «ты специалист по кинематографу»)
играет положительную роль, особенно для небольшой модели.

**Первый тест RAG.** На запрос про Роберта Де Ниро модель по-прежнему
выдала два несуществующих названия — потому что в текстах отзывов не
всегда упоминается название фильма, и модель не может сопоставить отзыв с
конкретным названием.

**Улучшение 1: добавление названия фильма в контекст.** В контекст
добавили название фильма из метаданных (а не только текст отзыва).
Результат резко улучшился — модель стала корректно перечислять фильмы.
Единственное изменение — добавление названия в контекст, сам поиск не
менялся.

**Улучшение 2: добавление реранкера.** Реранкер на основе кросс-энкодера
(`HuggingFaceCrossEncoder` из `langchain_community`). Число кандидатов
базового поиска увеличено с 5 до 50; для каждой пары (запрос, чанк)
считается скор кросс-энкодером, документы сортируются по убыванию скора,
берутся top-10 для контекста генератора. После добавления реранкера
дублирующийся ранее в выдаче отзыв стал встречаться только один раз.

**Тест: «Какой фильм выиграл больше всего Оскаров?»** Модель ответила
корректно — «Титаник» (11 Оскаров), опираясь на отзыв, где это явно
упоминалось.

**Тест: «Сколько лет Тому Холланду?»** Модель корректно не дала ответа,
поскольку в отзывах о фильмах такой информации нет, а промт ограничивает
модель отвечать только на основе контекста — механизм ограничения
галлюцинаций сработал как задумано.

**Улучшение 3: перефразирование запроса (multi-query).** Исходный запрос
переформулируется моделью в несколько (в примере — 3) разных вариаций
через отдельный промт («напиши N разных вариаций вопроса пользователя для
того, чтобы по ним получить релевантные документы из векторной базы
данных... не пиши ответ на сам вопрос»). Для каждой вариации отдельно
выполняется поиск, найденные чанки объединяются в общий контекст. Тест на
запросе «посоветуй лёгкую комедию» показал, что подход не сильно помог —
среди рекомендаций оказались явно не комедии. Ограничение — объём
доступного контекста ограничен размером контекстного окна выбранной
(небольшой) модели (в примере — 512 токенов).

**Улучшение 4: фильтрация по метаданным.** Векторные базы данных обычно
поддерживают встроенную фильтрацию по ключам метаданных (в примере — год
выхода фильма, в Qdrant — через функцию задания диапазона). Тест без
фильтра («список 5 лучших мелодрам 80-х») дал явную галлюцинацию — ни
один фильм не подходил ни по десятилетию, ни по жанру. Тест с фильтром по
годам (1980-е) дал заметное улучшение — все фильмы оказались из нужного
десятилетия, хотя соответствие жанру всё ещё было не идеальным.

## Заключение

Базовую RAG-систему на реальных данных (поиск + генератор) можно собрать
буквально за одно занятие, без серьёзных сложностей. Но при дальнейшем
тестировании системы разными вопросами быстро всплывают нюансы — где-то
модель галлюцинирует, где-то не даёт ответ, хотя могла бы. Это запускает
долгий, итеративный процесс улучшения, который может занимать уже не
часы, а недели: результат сильно зависит от данных, способа нарезки на
чанки, добавляемой метаинформации, выбора модели и т. д. Единого решения,
которое «однажды почини — и готово», не существует. При этом большинство
разобранных техник улучшения — не сверхсложные инженерные решения, а
относительно небольшие, доступные приёмы, которые вполне можно
реализовать в одиночку, без большой команды.
